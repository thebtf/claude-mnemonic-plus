# Engram v4.4.0 — Phase B Loom Integration

> **Published:** 2026-04-16 · **Tag:** [`v4.4.0`](https://github.com/thebtf/engram/releases/tag/v4.4.0) · **Main HEAD:** `ea47bfc`

Second tenant lands on the Phase A modular daemon framework. v4.4.0 ships
[aimux/loom v0.1.0](https://github.com/thebtf/aimux/tree/main/loom) as a full
EngramModule with 4 MCP tools, a shell-out CLI worker, OpenTelemetry metrics,
and a daemon-side bridge that consumes real server-driven project removal
events.

Plugin users get **4 new MCP tools** for background task submission and the
daemon now tears down project-scoped state automatically when a project is
removed from the engram-server dashboard.

## Highlights

### Loom tenant (`internal/handlers/loom/`)

- **EngramModule + 5 optional capabilities**: Init / Shutdown /
  ProjectLifecycle / ProjectRemovalAware / Snapshotter all implemented
  against a narrow `loomEngine` interface with compile-time assertion
  against `*loom.LoomEngine`.
- **4 new MCP tools** via `ToolProvider`:
  - `loom_submit` — enqueue a task with typed payload
  - `loom_get` — fetch task state + result by ID
  - `loom_list` — list tasks with status filtering
  - `loom_cancel` — cancel a running task
- **CLI worker** — registered via `RegisterWorker(WorkerTypeCLI, …)`,
  shells out to allowlisted binaries (`codex`, `claude`, `aimux`) with
  per-task argv + stdin piping. Context cancellation kills the real
  subprocess (not just the shell wrapper) via `sh -c "exec …"`.
- **Non-blocking task event forwarding** — `loom.EventBus` subscriptions
  fan out into `notifications/loom/task_event` MCP notifications on the
  owning session.

### Framework upgrades in support of loom

- **`obs.MeterFor(name)`** helper — module-scoped OTel Meter factory that
  produces per-module instruments without changing the `ModuleDeps` struct.
  loom uses it to expose task-queue depth, worker concurrency, and job
  duration histograms on the same OTel pipeline as the dispatcher metrics.
- **`Dispatcher.ConnectedProjectIDs()`** — read-only snapshot of the
  live session set (populated by `OnProjectConnect` / `OnProjectDisconnect`).
  Used by the serverevents bridge to build the `SyncProjectState` heartbeat
  request. D18 doc revised from "dispatcher is stateless" to "no mutex in
  the hot HandleRequest path": `tracked sync.Map` lives alongside the
  existing `draining atomic.Bool`, both outside the tool-dispatch hot
  path.

### Server-side: project lifecycle proto (FR-6/7/8)

- **New gRPC RPCs** in `proto/engram/v1/engram.proto`:
  - `SyncProjectState(ClientId, LocalProjectIds) → (Removed, Unknown)`
    — unary reconciliation used by daemon heartbeats.
  - `ProjectEvents(ClientId) → stream ProjectEvent` — persistent server-
    streaming RPC that pushes `PROJECT_EVENT_TYPE_REMOVED` events in real
    time.
- **Soft-delete with 30-day retention** (FR-8 Q3): `DELETE /api/projects/{id}`
  marks the row as `removed_at` + `removed_by` + `remove_reason`, and a
  background reaper worker hard-deletes after the window expires. Migration
  `082_projects_lifecycle.sql` adds the columns + indexes.
- **`projectevents.Bus`** — in-process fan-out from the REST handler to all
  connected gRPC stream subscribers, with per-subscriber buffered channels.

### Daemon-side: serverevents bridge (FR-9/10)

- **`internal/handlers/serverevents.Bridge`** — two concurrent delivery
  paths:
  1. `runEventStream` — persistent `ProjectEvents` RPC with exponential
     backoff (1 s → 60 s cap, reset on each successful open).
  2. `runSyncTicker` — 60 s ± 5 s jittered `SyncProjectState` heartbeat
     as a safety net for events missed during a stream drop.
- **256-entry LRU dedup** keyed on `(event_type, project_id)` so the same
  removal arriving via both paths fires `OnProjectRemoved` exactly once.
- **Panic isolation** — every handler runs under `defer+recover` so a
  crashy module cannot break fan-out to the rest (FR-15 applied to the
  bridge path).
- **URL hardening** — `parseGRPCAddr` rejects empty hosts with a clear
  error, `isHTTPS` parses the URL instead of matching a prefix, `safeURL`
  returns a placeholder on parse failure to keep credentials out of logs.
- **Owned gRPC connection** is closed on `Stop()` so the bridge releases OS
  resources on daemon shutdown.

## Follow-ups

- **ModuleDeps.Meter** as a first-class field is deferred to a future
  framework pass. v4.4.0 uses the `obs.MeterFor` helper as a low-risk
  Option B — modules pull the meter directly instead of receiving it
  in their `ModuleDeps` on Init.
- **Thinker / investigator / orchestrator worker types** — only
  `WorkerTypeCLI` ships in v4.4.0. The other worker types are
  stubbed pending domain work in `aimux/loom@v0.2.0`.
- **Windows named-pipe control socket** still deferred — the bridge's
  graceful-restart semantics work over the Unix-socket path today and
  the Windows path remains a `WARN` no-op.
